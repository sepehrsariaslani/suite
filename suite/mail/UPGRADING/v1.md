# Frappe Mail v1 Upgrade Guide (from v0.x.x)

This guide covers the upgrade path to Frappe Mail v1.x.x.

v1.x.x includes breaking changes required for compatibility with Stalwart 0.16.x:

- New Stalwart management flow based on Stalwart 0.16.x.
- Updated domain and member management flows in the Admin Dashboard.
- Mailing Lists temporarily removed and planned to return with a new flow.

## Before You Start

1. Read Stalwart's official upgrade guide for 0.16.x:
   https://github.com/stalwartlabs/stalwart/blob/main/UPGRADING/v0_16.md
2. Take a full backup or VM snapshot before making any changes.
3. Plan downtime for the upgrade window.
4. Confirm you can restore both:
   - Frappe site database.
   - Stalwart storage/data path.

## Storage Notes

- If your Stalwart backend uses external services (for example MySQL/MariaDB, S3, Redis), migration is usually straightforward.
- If you use embedded/local storage backends such as RocksDB or SQLite, review path/layout changes in Stalwart 0.16.x carefully and migrate those paths before migrating the mail app.

## Recommended Upgrade Order

1. Update the Mail app to v1.x.x.
2. Run bench migrations.
3. Configure Mail Cluster storage for the Stalwart 0.16.x deployment.
4. Set `server_url`, `username`, and `password` in Mail Settings.
5. Upgrade and migrate Stalwart to 0.16.x.
6. Regenerate account app passwords in Frappe Mail (required).
7. Validate domain, member, and mail flows.

## Step 1: Update the Mail App

Update the Mail app to the target v1.x.x tag in your deployment workflow.

## Step 2: Run Bench Migrate

From your bench directory:

```bash
bench --site <site-name> migrate
```

## Step 3: Configure Mail Cluster Storage

Before Stalwart migration, configure Mail Cluster storage backends for the Stalwart 0.16.x deployment model.

At minimum, verify:

- Store types and paths are correct for your environment.
- External dependencies (for example object storage/SQL/Redis) are reachable.
- You know which nodes/hosts will receive the upgraded deployment.

## Step 4: Configure Mail Settings Credentials

Ensure these values are configured in Mail Settings (preferred) or under the `mail` key in `site_config.json`:

- server_url
- username
- password

## Step 5: Upgrade and Migrate Stalwart

Follow Stalwart's official [guide](https://github.com/stalwartlabs/stalwart/blob/main/UPGRADING/v0_16.md) and finish all Stalwart-side migration tasks first.

Treat Step 5 as complete only when:

- Stalwart is running on 0.16.x.
- Admin credentials are known and working.
- Existing mail data is accessible.

### Stalwart 0.16.x Migration Runbook

Use this sequence for a controlled migration.

1. Export and convert data from the existing Stalwart instance.

```bash
mkdir -p backup
cd backup

curl -fLO https://raw.githubusercontent.com/stalwartlabs/stalwart/refs/heads/main/resources/scripts/migrate_v016.py

python3 -m venv .venv
source .venv/bin/activate
pip install requests urllib3

python migrate_v016.py dump \
    --url https://mail.example.com \
    --username frappe \
    --password admin-password \
    --settings settings.json \
    --principals principals.json

python migrate_v016.py convert \
    --settings settings.json \
    --principals principals.json \
    --config config.json \
    --output export.json
```

2. SSH to the target mail host and stop the currently running Stalwart container in `/opt/stalwart/<hostname>/`.
3. Take a physical backup of the Stalwart data directory before proceeding.
4. Trigger deploy from the Mail Server document so the new Stalwart version and layout are prepared.
5. Stop the newly deployed container before importing data.

```bash
cd /etc/stalwart/<hostname>/ && docker compose down
```

6. For RocksDB or SQLite only:
   - Clear files from `/data` in the new container volume.
   - Copy data files from backup, excluding any `LOCK` file.
7. Set `STALWART_RECOVERY_MODE=1` in `stalwart.env`.
8. Start Stalwart in recovery mode:

```bash
docker compose up
```

9. Import the converted export using `stalwart-cli` (`export.json`).

```bash
stalwart-cli --url http://mail.example.com:8080/ --user frappe --password admin-password apply --file export.json
```

10. Exit the container shell/session after import completes.
11. Set `STALWART_RECOVERY_MODE=0` in `stalwart.env`.
12. Start Stalwart normally (detached):

```bash
docker compose up -d
```

13. Verify Stalwart health checks, admin login, and mailbox accessibility before moving to Step 6.

## Step 6: Regenerate App Passwords for Existing Accounts

Stalwart's migration does not carry over existing app passwords used by Frappe Mail accounts.

Run this in bench console after Stalwart 0.16.x is fully up and credentials are configured:

```python
import frappe
from frappe import _
from mail.stalwart import create_app_password
from mail.utils.user import is_jmap_configured

settings = frappe.db.get_all("User Settings", {}, ["name", "user"])

if not settings:
    frappe.throw(_("No user settings found to update app password."))

for user_settings in settings:
    if not is_jmap_configured(user_settings["user"]):
        print(
            _("Skipping user {0} as JMAP is not configured.").format(
                user_settings["user"]
            )
        )
        continue

    print(_("Updating app password for user {0}").format(user_settings["user"]))

    try:
        app_password = create_app_password(user_settings["user"])

        user_settings_doc = frappe.get_doc("User Settings", user_settings["name"])
        user_settings_doc.app_password = app_password
        user_settings_doc.save(ignore_permissions=True)

    except Exception as error:
        frappe.log_error(
            title=_("Error updating app password for account {0}").format(
                user_settings["name"]
            ),
            message=str(error),
        )

frappe.db.commit()

```

Why this is manual:

- The patch is intentionally not automatic because some sites may migrate the app before Stalwart is upgraded to 0.16.x.
- Running it only after Stalwart is ready prevents partial/broken updates.

## Step 7: Validate the Upgrade

Use this checklist after migration:

1. Open Admin Dashboard and validate:
   - Domains list and domain details pages.
   - Members and Invites flows.
2. Test mailbox actions:
   - Send mail.
   - Receive mail.
   - Open existing threads.
3. Confirm user login and account access for a sample of existing users.

## Deployment Notes for Server Deployment Users

If you previously deployed using Mail Cluster and Mail Server documents:

- Recommended: recreate deployment documents to match the new v1/Stalwart 0.16.x model.
- Alternative: reuse existing documents only after carefully reviewing storage and bootstrap config.

At minimum, verify:

- Mail Cluster storage configuration is valid for 0.16.x.
- Mail Server bootstrap/config values are up to date.

## Rollback Guidance

If validation fails:

1. Stop upgrade activity.
2. Restore the pre-upgrade Frappe database and Stalwart data snapshot.
3. Roll back to previous app/stalwart versions.
4. Re-run the upgrade in a staging environment and resolve failures before retrying production.

## Known v1 Behavior Changes

- Domain and member creation flows have changed in the dashboard.
- Mailing Lists are temporarily unavailable in v1 and will return in a future release with a revised implementation.
